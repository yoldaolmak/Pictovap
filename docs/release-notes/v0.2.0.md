# Pictovap v0.2.0 Release Notes

This release establishes the public open-source core: credential-free demo, documented primitives, adapter model, tests, and contribution paths.

## Summary

Pictovap is an open-source visual finishing engine for content publishers. It turns unfinished articles into visually complete, rights-aware, publish-ready CMS pages.

This is the first public baseline release. It makes the core pipeline runnable locally without any credentials or external APIs and provides the documentation, test suite, and contribution paths needed for external evaluation.

## What Works

- The credential-free local demo runs the full pipeline end-to-end using mock/local candidates.
- Four core primitives are implemented and documented:
  1. **Visual Brief** — Analyzes article structure to generate image requirements.
  2. **Fit Score** — Evaluates candidates deterministically against contextual, technical, and licensing criteria.
  3. **Provenance Pack** — Maintains an audit trail for selected images (source, license, processing actions).
  4. **CMS Placement** — Produces CMS-agnostic placement instructions.
- The demo accepts custom Markdown articles and publisher profiles.
- Unit tests (99) pass, including a public-language guard test.
- CI runs smoke tests and demo verification on every push.

## How to Run the Demo

```bash
git clone https://github.com/yoldaolmak/Pictovap.git
cd Pictovap
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
pip install -e .
make demo
```

Or run with your own article. The demo outputs canonical JSON by default, but you can optionally generate an editor-readable Markdown report:

```bash
python -m pictovap.demo \
  --article path/to/your/article.md \
  --output my-plan.json \
  --report my-report.md
```

## What the Demo Does Not Do

- It does not make live calls to external model providers or paid APIs.
- It does not fetch live stock photos from Unsplash or DepositPhotos.
- It does not push actual content to a live CMS.
- It does not produce legally binding provenance guarantees.
- It relies entirely on mock assets and deterministic scoring to demonstrate the pipeline architecture reliably.

| Primitive | Purpose | Status |

| Visual Brief | Analyzes article to determine image requirements | Implemented |
| Fit Score | Deterministic candidate evaluation (contextual + quality + license) | Implemented |

## Adapter Status

| Adapter | Type | Status |
|---------|------|--------|
| Local folder | Image source | Implemented |
| Visual Memory DB | Image source | Implemented (Mac Photos index) |
| Unsplash | Image source | Reference implementation |
| DepositPhotos | Image source | Stub / planned |
| WordPress | CMS publisher | Strongest adapter, extracted from production usage |
| Ghost | CMS publisher | Not yet implemented |
| Strapi | CMS publisher | Not yet implemented |

WordPress is currently the strongest CMS adapter — it was extracted from real production use at a single publisher (dogfooding). Other CMS adapters are future work or reference/stub paths.

## Current Limitations

- The pipeline was extracted from a single publisher's production use. Some internal dictionary passing is still being migrated to strict typed primitives.
- The local demo uses deterministic mock/local candidates, not live sources.
- Only the WordPress CMS adapter is production-hardened; others are reference implementations.
- The default structural extraction handles English and Turkish articles reliably; other languages are untested.
- There is no external adoption yet. This release is a baseline for external evaluation.

## Known Risks

- **Provenance ≠ Legal Guarantee.** The Provenance Pack provides an audit trail of image sources and licenses. It is not a legal guarantee. Publishers remain responsible for verifying correct licensing for their jurisdiction.
- **Internal Namespace.** The internal Python package namespace remains `pictova` for backward compatibility, while the product name is `Pictovap`.
- **WordPress State Drift.** Using the WordPress adapter without maintaining persistent state for `PICTOVA_POST_MANIFEST_DIR` across deployments could lead to orphaned media blocks in the CMS.
- **No External Validation.** This project has not been validated by external contributors, downstream dependents, or third-party reviewers.

## Upgrade / Compatibility Notes

- **Product name:** Pictovap (always capitalized in documentation).
- **Python package / CLI:** The internal package directory (`src/pictovap/`) and legacy CLI command (`pictova`) remain lowercase for backward compatibility. A `pictovap` CLI alias is also available.
- **No breaking changes** from earlier internal versions — this is the first public baseline.
